1. Executive framing

Position Translator as a workflow service, not just a language widget

Azure AI Translator is a cloud based neural machine translation service that supports real time text translation, single file document translation, batch document translation, and deeper customization through custom models. For ABS, the most important question is not whether Translator can translate, but how it can fit into content operations while keeping maritime terminology consistent.

Customer friendly framing

  • Translator helps ABS move faster across languages without rebuilding the content pipeline.
  • It can serve lightweight use cases and enterprise workflows with the same underlying service family.
  • Glossaries give ABS a practical way to enforce preferred terminology before considering custom models.

Presenter note

A strong opening line is: “Translator gives you a spectrum of options, from quick single document translation to governed, batch document workflows with terminology control and enterprise security.”

2. How Azure AI Translator works

One service family, multiple paths depending on the content and workflow

The official Microsoft guidance separates Translator into text translation, document translation, and Custom Translator. Document translation itself supports both synchronous single file translation and asynchronous batch translation.

1

Choose the workflow

Real time text, single file document translation, or batch translation across many files.

2

Send content to Translator

Use REST APIs, SDKs, or Foundry for supported scenarios. Batch workflows use Blob Storage.

3

Apply control layers

Use glossary for term control. Use custom models when baseline plus glossary is still not enough.

4

Return translated output

Preserve document layout where supported, then pass translated output into review and publishing.

Synchronous document translation

Best for a single document and ideal for your demo. The request includes one file and can include an optional glossary. No Blob Storage account is required, and the translated file returns directly in the response.

Single file Optional glossary Fast demo path

Asynchronous batch document translation

Best for production scale. Source files live in Blob Storage, translation runs as a job, and outputs land in target containers. This is the better fit for large document sets or CMS export based workflows.

Many files Blob Storage based Operational scale

How to say it in the room

“For today’s demo I am using the synchronous single file flow because it is the cleanest way to show the effect of glossary control. For an ABS production workflow, the same service family also supports batch document translation at storage based scale.”

3. Key capabilities

Map the capability to the scenario

Not every translation need should be solved the same way. The value comes from choosing the simplest option that meets the accuracy and workflow need.

Capability What it does Best fit ABS message
Text translation Real time translation of text via API. Supports detect, transliterate, and dictionary operations. Apps, workflow automation, short text, live experiences Good for app experiences or content fragments, not the main PDF demo
Document translation, single file Translate one file and return the translated document directly. Can include a glossary. Demo, ad hoc file translation, simple user workflows This is the demo path you are using today
Document translation, batch Translate many files asynchronously while preserving structure and format. Large document sets, export based localization, repository scale jobs Best longer term option for governed ABS translation operations
Glossary Apply source to target terminology control during document translation. Known terminology, branded language, domain specific vocabulary Fastest way to improve ABS maritime terms without model training
Custom Translator Build custom neural translation systems for domain specific language, terminology, and style. When baseline plus glossary still falls short and enough data exists Potential future phase, not the first answer for this discussion
4. Accuracy and customization

Use the right control layer for the right accuracy problem

The customer should leave understanding that Azure AI Translator already delivers strong baseline translation, and that there are different levers for improving output depending on the nature of the issue.

Baseline neural translation

Best starting point for general language quality. This is the default system you compare against in the demo.

  • Useful for general meaning and speed to value
  • Good enough for many broad content scenarios
  • May not pick the preferred industry term every time

Glossary driven term control

Best when the challenge is preferred terminology. Microsoft guidance for document translation supports glossary files such as TSV, CSV, and XLIFF.

  • One to one source to target language translation
  • Useful for technical vocabulary, product names, and ambiguous words
  • Case sensitive by default, so source matching matters

Custom Translator

Best when the issue is larger than a term list, such as domain style, sentence patterns, or recurring phraseology.

  • Requires curated training data and governance
  • More effort than a glossary
  • Good follow on phase if glossary alone is not enough

Important demo framing

Glossary improves terminology control. It does not guarantee that every other phrase in the document becomes perfect. That framing keeps the audience focused on the right success criterion.

Use your current ABS proof points

  • loose gear → accesorios de izado
  • sheaves → roldanas
  • Repeated technical terms are where the glossary story lands best

What not to overclaim

  • Do not say the glossary solves every translation issue
  • Do say it makes the important domain terms more consistent
  • Do say it is the fastest practical tuning lever for this use case
5. Scalability and enterprise considerations

Show that this can start small and still fit enterprise expectations

The service is not just for experiments. Microsoft documents supported enterprise patterns for authentication, storage authorization, network isolation, data residency, logging, and secure deployment.

Scalability

  • Batch document translation supports numerous files and larger workflows.
  • Batch jobs use source and target Blob Storage containers.
  • Layout and document structure are preserved for supported formats.

Security and access

  • Text translation supports resource key, bearer token, and Microsoft Entra ID authentication.
  • Batch document translation can use SAS tokens or managed identities to reach storage.
  • Azure Key Vault is the right place for secret handling and key rotation.

Network and compliance

  • Private endpoints, firewalls, and region specific endpoints are available patterns.
  • Data residency depends on the Azure region used for the resource.
  • Containers are an option for more isolated or offline scenarios.

Enterprise line to use

“This can begin as a targeted language service for a single workflow and still be deployed with the security, identity, storage, and network controls you would expect in an enterprise Azure environment.”

Human oversight

Microsoft guidance also supports the practical reality that sensitive translations should include human review. That is especially useful for customer facing, regulated, or safety relevant content.

6. ABS fit and recommendation

Recommend the simplest useful path first

Based on the ABS scenario discussed so far, the most pragmatic path is to start with standard Translator plus glossary control, prove value with real content, then expand into batch workflows and only consider custom models if the residual quality gap justifies it.

Phase 1: prove immediate value

Goal: show that ABS terminology can be controlled without a long setup effort.

  • Use single file document translation
  • Apply ABS maritime glossary
  • Demonstrate before and after on selected technical pages

Phase 2: operationalize

Goal: move from demo to repeatable workflow.

  • Shift to batch document translation where appropriate
  • Use Blob Storage, managed identity, and review workflow
  • Connect outputs to downstream publishing or review steps

Phase 3: deepen accuracy

Goal: address residual domain specific issues that a glossary cannot solve alone.

  • Evaluate Custom Translator if enough curated data exists
  • Use human review for high impact content classes
  • Expand terminology governance over time
7. Demo talk track

Tee up the demo so the customer watches for the right signal

Your current excerpt is stronger now because it is more glossary centric. The story should emphasize repeated technical term control, not a claim that every sentence becomes perfect.

Recommended setup narrative

  1. “I am using the same source excerpt, same target language, and same service in both runs.”
  2. “Run one is baseline Translator with no glossary.”
  3. “Run two adds a domain glossary for ABS terminology.”
  4. “The point is to show how preferred terms become more consistent.”

Recommended control surface in code

Use a glossary path parameter rather than a simple on off flag. It is cleaner and more reusable.

glossaryPath = null glossaryPath = "glossary/glossary_v1-en-es.tsv"

Current terms to point out live

  • loose gear
  • sheaves
  • proof testing
  • wire ropes

What to say after the reveal

“This is the practical value of a glossary. We are not retraining the whole system here. We are adding a governed terminology layer so the terms that matter to ABS are translated the way ABS wants them translated.”

Optional transition into next discussion

“If this approach looks promising, the next conversation is less about whether translation is possible and more about where ABS wants to apply it first: ad hoc files, repeatable document sets, or a broader content workflow.”
